3.3 Chemistry

In this field of the setup file the user can define all the details related with the kinetic scheme, i.e. with the configuration of the LoKI-C module. The code below shows the section of the 'default_lokibc_setup.in' file that handles this configuration.

% --- configuration of the  heavy-species for the kinetic scheme ---
chemistry:
  isOn:  true
  convergenceParameter: electronDensity     %  electronDensity or dischargeCurrent or dischargePowerDensity
  thermalModel:
    isOn: true                              % true or false (to activate or deactivate the thermal model)
    boundary: wall                          % wall or external (location of the model boundary condition)
    wallFraction: 1                         % fraction of energy released at the wall that returns
                                            % to the gas/plasma volume (check fw parameter in the documentation)
    intConvCoeff: 120/1.6021766208e-19      % in eV s-1 m-2 K-1
                                            % (internal convection coefficient of the cylindrical wall)
    extConvCoeff: 15/1.6021766208e-19       % in eV s-1 m-2 K-1
                                            % (external convection coefficient of the cylindrical wall)
    includeJouleHeating: false
  chemFiles:
    - Nitrogen/nitrogen_mod.chem
  checkChemFilesStoichiometry: true         % true or false
                                            % (to activate or deactivate stoichiometry check in chem files)
  gasProperties:                            % inherited from the configuration of the electron kinetics
    mass: Databases/masses.txt
    heatCapacity:                           % in eV K-1 mol-1 (heat capacity functions at constant pressure)
      - N2 = nitrogenHeatCapacity
      - N = nitrogenHeatCapacity
      - N3 = nitrogenHeatCapacity
      - N4 = nitrogenHeatCapacity
    thermalConductivity:                    % in eV s-1 m-1 K-1 (thermal conductivity functions)
      - N2 = nitrogenThermalConductivity
      - N = nitrogenThermalConductivity
      - N3 = nitrogenThermalConductivity
      - N4 = nitrogenThermalConductivity
    inFlowFraction:
      - N2 = 1
    stateProperties:
      energy:
        - N2(X,v=*) = morseOscillatorEnergy
      statisticalWeight:
        - N2(X,v=*) = 1.0
      population:
        - N2(X) = 1.0
        - N(4S) = 1.0
      inFlowPopulation:
        - N2(X) = 1
        - N2(X,v=0) = 1
      reducedDiffCoeff:
        - N2(A3Su+) = generalizedTemperatureDependentCoeff@5.0e20/sqrt(300),gasTemperature,0.5
        - N2(B3Pg) = generalizedTemperatureDependentCoeff@5.0e20/sqrt(300),gasTemperature,0.5
        - N2(C3Pu) = generalizedTemperatureDependentCoeff@5.0e20/sqrt(300),gasTemperature,0.5
        - N2(a1Pg) = generalizedTemperatureDependentCoeff@5.0e20/sqrt(300),gasTemperature,0.5
        - N2(a'1Su-) = generalizedTemperatureDependentCoeff@5.0e20/sqrt(300),gasTemperature,0.5
        - N2(w1Du) = generalizedTemperatureDependentCoeff@5.0e20/sqrt(300),gasTemperature,0.5
        - N2(+,X) = generalizedTemperatureDependentCoeff@1.7e20/sqrt(273),gasTemperature,0.5
        - N2(+,B2Su+) = generalizedTemperatureDependentCoeff@1.7e20/sqrt(273),gasTemperature,0.5
        - N3(+,X) = generalizedTemperatureDependentCoeff@1.7e20/sqrt(273),gasTemperature,0.5
        - N4(+,X) = generalizedTemperatureDependentCoeff@1.7e20/sqrt(273),gasTemperature,0.5
        - N(2D) = generalizedTemperatureDependentCoeff@6.4e20/sqrt(300),gasTemperature,0.5
        - N(2P) = generalizedTemperatureDependentCoeff@5.2e20/sqrt(300),gasTemperature,0.5
        - N(+,gnd) = generalizedTemperatureDependentCoeff@1.7e20/sqrt(273),gasTemperature,0.5
      reducedMobility:
        - N2(+,X) = 1.87e-4*101325/(273*1.38064852e-23)
        - N2(+,B2Su+) = 1.87e-4*101325/(273*1.38064852e-23)
        - N3(+,X) = 1.87e-4*101325/(273*1.38064852e-23)
        - N4(+,X) = 2.33e-4*101325/(273*1.38064852e-23)
        - N(+,gnd) = 1.87e-4*101325/(273*1.38064852e-23)
      % flowBarrier:        % flow barrier (factor between 0-1; by default is set = 1)
      %   - N2(A3Su+) = 0.05
      %   - N2(X,v=0) = 0.1
      iterationSchemes:
        % pressureRelError: 1e-5
        pressureMaxIterations: 1000
        % neutralityRelError: 1e-5
        neutralityMaxIterations: 500
        % globalRelError: 1e-4
        globalMaxIterations: 100
      timeIntegrationConf:
        odeSolver: ode15s
        dischargeTime: 10
        postDischargeTime: 0
        odeSetParameters:   % optional parameters that can be sent to the odeSolver
          MaxStep: 0.1
          RelTol: 5e-4
%          AbsTol: 1e-10

IMPORTANT NOTES:

  • The isOn allows the user to activate (true) or deactivate (false) the chemistry simulations of LoKI-C.
  • LoKI-C supports three types of experimental input conditions for steady-state simulations, through convergenceParameter: electronDensity, dischargeCurrent and dischargePowerDensity.

    When selecting electronDensity, the quasi-neutrality cycle of the code iterates over E/N to satisfy the neutrality condition ∑k=ionnk=ne, for given electron density ne.
    When selecting dischargeCurrent, the quasi-neutrality cycle of the code iterates over E/N and ne to satisfy the neutrality condition ∑k=ionnk=ne, for given electron density ne, for given discharge current I=eneμeEπR2.
    When selecting dischargePowerDensity, the quasi-neutrality cycle of the code iterates over E/N and ne to satisfy the neutrality condition ∑k=ionnk=ne, for given discharge power density P/V=eΘEne.

    For steady-state simulations, the user MUST define in the workingConditions the values of one of the convergence parameters: electronDensity, dischargeCurrent or dischargePowerDensity.
    For quasi-stationary simulations and post-discharge simulations, the user MUST define in the Working Conditions the value of the electronDensity, which is taken as an initial value in these time-dependent simulations. (see section 2.5). For more details please check the Reference Manual [11])
  • In thermalModel the user can specify relevant details for the configuration of the thermal model:
    • isOn allows the user to activate (true) or deactivate (false) the thermal model calculations in LoKI-C.
      When the thermal model is activated the user MUST define in the workingConditions the gasTemperature and one of the boundary condition parameters: wallTemperature for the wall boundary condition or extTemperature for the external boundary condition (see section 3.1).
    • boundary allows the user to specify the boundary condition of the thermal model, wall or external (see Reference Manual [11]);
    • wallFraction corresponds to the parameter fw (see Reference Manual [11]), representing the fraction of the enthalpy-exchange in wall-reactions that returns to the gas/plasma volume.
      Note that this parameter is automatically SET TO ZERO by LoKI, for the external boundary condition.
    • intConvCoeff corresponds to the convection parameter hint > 0, in the inner region near the tube wall.
      To choose hint ≫ 1 the user should set intConvCoeff = inf.
    • extConvCoeff corresponds to the convection parameter hext > 0, in the region outside the cylindrical wall (used with the external boundary condition only).
      To choose hext ≫ 1 the user should set extConvCoeff = inf.
    • The includeJouleHeating allows the user to activate (true) or deactivate (false) a thermal source term corresponding to the total power transferred from the applied electric field to the electrons, as calculated by LoKI-B.
      Note that when this option is enabled, all other source terms in the thermal balance equation are automatically SET TO ZERO.
  • The chemFiles allows the user to specify the text files with the description of the kinetic scheme to adopt, namely the reactions, the gases and the states of those gases to be included in the simulations; see section 4.3 for more information.
  • The checkChemFilesStoichiometry allows the user to activate (true) or deactivate (false) a stoichiometry check in the chemistry files with the description of the kinetic scheme.
  • For quasi-stationary simulations, the chemistry configuration MUST include an extra field lookUpTables, with several subfields (see section 2.5)
    • power, swarm, rateCoeff, pointing to the corresponding lookup-table files;
    • lookUpMethod, which allows the user to choose the localField or localEnergy method to interpolate the electron parameters from the lookup tables (see more details in Reference Manual [11]).
    The Code shows an example of configuration for the parameter lookUpTables. 
    lookUpTables:
  power: Nitrogen_pulsed/lookUpTablePower.txt
  swarm: Nitrogen_pulsed/lookUpTableSwarm.txt
  rateCoeff: Nitrogen_pulsed/lookUpTableRateCoeff.txt
  lookUpMethod localField  % localField or localEnergy
    Note the following:
    • the header names in the lookup tables MUST correspond exactly to
      • RedField(Td) in all lookup tables;
      • EleTemp(eV), RedMob((msV)^-1), RedDiff((ms)^-1) and DriftVelocity(ms^-1) in the lookUpTableSwarm.txt;
      • RID_ine(m^3s^-1) and RID_sup(m^3s^-1), with ID= 1, 2, 3, · · · in the lookUpTableRateCoeff.txt;
      • PowerField(eVm^3s^-1), PwrElaNet(eVm^3s^-1), PwrCARNet(eVm^3s^-1), PwrVibNet(eVm^3s^-1) and PwrRotNet(eVm^3s^-1) in the lookUpTablePower.txt;
    • the lookup table lookUpTableRateCoeff.txt begins with a header listing the ID and the Description of the reactions associated with the rate coefficients and extra rate coefficients provided in the table. The electron-impact reactions specified in this header MUST correspond exactly to those defined in the chemistry file.
    • the different lookup tables must be generated consistently, i.e., using the same number and values of the reduced electric field.
  • In gasProperties the user can specify the relevant properties of the different gases contributing with heavy species to the chemistry simulations, such as
    mass
    fraction
    harmonicFrequency
    anharmonicFrequency
    ...
    This field is equivalent, in every aspect, to the one described in section 3.2, hence all the considerations mentioned there apply here. Note, however, the existence of several specific properties in LoKI-C:
    • heatCapacity allows defining the molar specific heat at constant pressure (Ck ; see Reference Manual [11]) of the different (atomic/molecular) gases involved in the simulations;
    • thermalConductivity allows defining the thermal conductivities (λk; see Reference Manual [11]) of the different (atomic/molecular) gases involved in the simulations;
    • inFlowTraction allows defining the inflow fractions of the different gases in the mixture. This parameter should add to 1, when summed over all gases used in the simulations.
    The first two properties (to be evaluated in SI units) are mandatory if the thermal model is activated. The last property is mandatory if the totalSccmInFlow defined in the workingConditions, is different from zero.
    Please note that, when running both LoKI-B and LoKI-C, the species present in LoKI-C may or may not coincide with the species present in LoKI-B. Furthermore, when LoKI-B and LoKI-C share the same species, their properties may or may not coincide in both codes. For these reasons, the following rules apply to this field:
    • When running both modules (LoKI-B and LoKI-C) the gasProperties subfield of the chemistry field is optional. By default, the properties of the gases present in the LoKI-C simulation are inherited from those dened in the electronKinetics field. However, the user can fill the gasProperties subfield of the chemistry field if they needs to: (i) override the values provided in the electronKinetics configuration, for the LoKI-C calculations; and/or (ii) provide values for the properties of gases that are not present in the LoKI-B calculations (e.g., heatCapacity, thermalConductivity) and inFlowFraction.
    • When running LoKI-C as standalone tool the gasProperties subfield of the chemistry field is mandatory. In particular, it is mandatory to provide the fraction property for each gas present in the simulations.
  • In stateProperties the user can specify the properties of the states of the different gases contributing with heavy species to the chemistry simulations. The properties available are
    energy
    statisticalWeight
    population
    reducedDiffCoeff
    reducedMobility
    flowBarrier
    This field is equivalent, in every aspect, to the one described in section 3.2, hence all the considerations mentioned there apply here. Note, however, the existence of several new properties that are important for the implementation in LoKI-C of the transport losses. In particular
    • inFlowPopulation allows defining the inflow populations of the different parent-states of the inflow gases in the mixture. The values of the different inflow populations should be properly normalized to 1, when summed over all sibling states.
      This property is mandatory if the totalSccmInFlow, defined in the workingConditions, is different from zero;
    • flowBarrier is a parameter between 0-1 (with default value 1), which allows controlling the outflow of each state of the different gases in the mixture (see Reference Manual [10]).
      This property is used only if the totalSccmOutFlow is different from zero, as defined in the working Conditions;
    • reducedDiffCoeff allows defining the reduced diffusion coefficients (Dki, parent of ki for neutrals and Dip for ions; see Reference Manual [11]) of the different (atomic/molecular) states involved in the simulations;
    • reducedMobility allows defining the reduced mobilities (μip for ions; see Reference Manual [11]) of the different (atomic/molecular) states involved in the simulations.
    The last two properties (to be evaluated in SI units) are mandatory for those states exhibiting transport losses, and can be specified by using the "direct input" method
       - N2(+,X) = 1.87e-4*101325/(273*1.38064852e-23)
    or the "property function" method 
       - N2(+,X) = generalizedTemperatureDependentCoeff@1.7e20/sqrt(273),gasTemperature,0.5
    Note further that, when running both LoKI-B and LoKI-C, the species present in LoKI-C may or may not coincide with the species present in LoKI-B. Furthermore, when LoKI-B and LoKI-C share the same species, their properties may or may not coincide in both codes. For these reasons, the following rules apply to this field:
    • When running both modules (LoKI-B and LoKI-C) the stateProperties subfield of the chemistry field is optional. By default, the properties of the gases present in the LoKI-C simulation are inherited from those defined in the electronKinetics field. However, the user can fill the stateProperties subfield of the chemistry field if he/she needs to: (i) override the values provided in the electronKinetics configuration, for the LoKI-C calculations; and/or (ii) provide values for the properties of gases that are not present in the LoKI-B calculations (e.g., reducedDiffCoeff, reducedMobility and inFlowPopulation).
    • When running LoKI-C as standalone tool the stateProperties subfield of the chemistry field is mandatory. In particular, it is mandatory to provide the population property for each state present in the simulations.

    Note finally that, in quasi-stationary simulations the user MUST define the population of the different ion states consistent with the value used for electronDensity in the workingConditions, such that (∑kχk∑i=ionδki)N=ne (see section 2.5).

    In simulations using the LoKI-GM, the user can define different initial populations for LoKI-B (on the electronKinetics) and LoKI-C (on the chemistry, see section 3.3). In this case, the following should be noted. At the beginning of the simulation, LoKI checks the normalization of the populations of both, the states considered as targets of electron-scattering mechanisms (termed LoKI-B states) and the states included in the chemistry scheme (termed LoKI-C states). After the evaluation of the densities by the chemistry model, and before returning to the electron kinetics calculations, LoKI renormalizes the populations of LoKI-B states to the total density of the states that are simultaneously LoKI-B states and LoKI-C states. Therefore:

    • in the electronKinetics, all LoKI-B states with non-zero populations should be also LoKI-C states. The exception are complete distributions of sibling states (e.g. the rotational distribution of a vibrational ground-state), which can be targets of electron-scattering mechanisms, being absent from the chemistry scheme;
    • in the electronKinetics, non-zero populations are allowed for LoKI-B states only. In response to this requirement, the distribution of LoKI includes some property functions with truncated distributions (e.g. boltzmannPopulationRotationalCutoff and boltzmannPopulationVibrationalCutoff, see section 5.1), which allow the user to define normalized populations for a subset of the states involved in the complete distribution.
    If the previous items are not taken into account by the user, the codes produces an error.

    The user is advised to initialize the populations of LoKI-B states and LoKI-C states adopting distributions collapsed to the corresponding ground-states.

  • In iterationSchemes the user can specify the numerical details for the different cycles implemented in LoKI-C (pressure cycle, neutrality cycle and global cycle). For each cycle, the user can specify the following parameters
    • pressureRelError, which defines the convergence threshold for the pressure cycle. This is an optional parameter. By default, its value is set to 2⨉RelTol, with typical value 10⁻³;
    • pressureMaxIterations, corresponding to the maximum number of iterations of the pressure cycle. This is a mandatory parameter when the pressure cycle is active, i.e. if the totalSccmOutFlow is different from the model ensureIsobaric, as defined in the workingConditions (see more details in Reference Manual [10]). The typical value is 1000;
    • neutralityRelError, which defines the convergence threshold for the neutrality cycle. This is an optional parameter. By default, its value is set to 2⨉RelTol, with typical value 10⁻³;
    • neutralityMaxIterations, corresponding to the maximum number of iterations of the neutrality cycle. This is a mandatory parameter for steady-state simulations (see section 2.5; for more details, see Reference Manual [10]). The typical value is 500;
    • globalRelError, which defines the convergence threshold for the global cycle. This is an optional parameter. By default, its value is set to 2⨉RelTol, with typical value 10⁻³;
    • globalMaxIterations, corresponding to the maximum number of iterations of the global cycle. This is a mandatory parameter for steady-state simulations (see section 2.5; for more details, see Reference Manual [10]). The typical value is 100.
    The user can deactivate any of the cycles by setting its maximum number of iterations to 1.
    Moreover, the neutrality cycle and the global cycle are deactivated when running LoKI-C as standalone tool (this behaviour might change in a future release of the code).
  • In timeIntegrationConf the user can specify the numerical details of the time integration for the rate balance equations with LoKI-C. The available subfields for the configuration of this parameter are:
    • odeSolver, specifying the ordinary differential equation (ODE) solver [20] to adopt in the time integration. This is a mandatory parameter. Presently, we strongly recommend the solver ode15s, because it has provided the best results in terms of accuracy and performance for the cases tested.
    • dischargeTime, corresponding to the final integration time of the rate balance equations with LoKI-C, for active discharges. This is a mandatory parameter for steady-state simulations, and also for post-discharge simulations that begin by running a steady-state case (see section 2.5; for more details, see Reference Manual [11]).
      Note that, in order to avoid numerical errors and preserve good performance, the dischargeTime should not be much longer than the time needed for the system to reach its steady-state situation. The typical value is 10 s.
      The dischargeTime can be set to zero provided that postDischargeTime is non-zero (see below). In this case, the discharge phase is skipped and LoKI-GM performs only the post-discharge simulation.
    • postDischargeTime, corresponding to the final integration time of the rate balance equations with LoKI-C, for a post-discharge. This is a mandatory parameter for post-discharge simulations.
      This parameter should be set to zero, to deactivate the study of a post-discharge (see section 2.5).
      Note that, in order to avoid numerical errors and preserve good performance, the postDischargeTime should not be too long. A typical value is 10⁻³ s.
    • odeSetParameters is an optional field where the user can define specific options for the solver selected in the field odeSolver. A detailed description of the different options can be found in the MATLAB® documentation [20]. Some examples of useful options are given below
      • MaxStep: is the maximum value for the step of the advancing parameter with the odeSolver chosen. The typical value is 0.1.
      • RelTol: is the relative tolerance adopted in the odeSolver chosen. The typical value is 5⨉10⁻⁴ (the default MATLAB® value is 10⁻³).
      • AbsTol: is the absolute tolerance adopted in the odeSolver chosen. The typical value is 10⁻¹⁰
      Inexperience users should not change the values chosen for these parameters, since their wrong definition will degrade the performance of the code and the quality of the calculations.
